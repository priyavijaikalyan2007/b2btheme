# Contributing to Enterprise Bootstrap Theme

Thank you for your interest in contributing to the Enterprise Bootstrap Theme. We appreciate your time and effort in helping improve this project.

## About This Project's Development Model

This project serves a dual purpose:

1. **Production-quality software** — A comprehensive Bootstrap 5 enterprise theme with 90+ custom components, built to high standards of performance, accessibility, and maintainability.
2. **Reference implementation for AI-agent-assisted development** — A showcase of how coding agents can build and maintain a large-scale, production-grade codebase with consistent architecture, coding style, and documentation.

All code changes are implemented by coding agents under the guidance and review of the project maintainers. This approach ensures strict adherence to the project's architectural decisions, coding standards, naming conventions, and documentation requirements. Because of this development model, **we do not accept pull requests**.

Contributions are welcome and valued in the forms described below.

## How to Contribute

### Reporting Bugs

If you find a bug, please open a [GitHub issue](../../issues/new) with the following information:

- **Description** — A clear, concise description of the bug.
- **Steps to Reproduce** — Detailed steps to reproduce the behavior.
- **Expected Behavior** — What you expected to happen.
- **Actual Behavior** — What actually happened.
- **Screenshots** — If applicable, add screenshots to help illustrate the problem.
- **Environment** — Browser, OS, and any relevant version information.
- **Component** — Which component is affected (e.g., Toast, DataGrid, DiagramEngine).

Well-written bug reports help us identify and fix issues quickly.

### Feature Requests

We welcome feature requests. Please open a [GitHub issue](../../issues/new) and include:

- **Problem Statement** — Describe the problem or need the feature would address.
- **Proposed Solution** — Your idea for how the feature could work.
- **Alternatives Considered** — Any alternative solutions or features you have considered.
- **Use Case** — A concrete example of when and how this feature would be used.
- **Component Scope** — Whether this applies to an existing component or would require a new one.

Feature requests are reviewed by the maintainers, prioritized, and implemented by coding agents when accepted.

### Suggesting Improvements

For general ideas, design discussions, or questions about the project's direction, please use [GitHub Discussions](../../discussions). This is the right place for:

- Architecture or design suggestions
- Questions about component behavior or API design
- Ideas for new component categories
- Discussion about coding patterns or best practices
- Feedback on documentation or developer experience

### Security Issues

If you discover a security vulnerability, please report it responsibly:

1. **Do NOT open a public issue** for security vulnerabilities.
2. Open a [GitHub security advisory](../../security/advisories/new) or send details via a private GitHub issue.
3. Include a clear description of the vulnerability, steps to reproduce, and the potential impact.
4. Allow reasonable time for the maintainers to address the issue before any public disclosure.

We take security seriously and will respond promptly to verified reports.

## Why We Don't Accept Pull Requests

This is not a reflection of the quality of your work. Our development model relies on coding agents to implement all changes because:

- **Consistency** — Agents enforce the project's Allman brace style, function length limits, DOM helper patterns, logging conventions, and 75+ architectural decisions documented in the knowledge base.
- **Architecture alignment** — Every change is validated against the project's component patterns, SCSS variable conventions, TypeScript module structure, and build pipeline.
- **Documentation** — Agents automatically maintain component READMEs, the component index, the master component list, and the structured knowledge base alongside every code change.
- **Quality gates** — The agent workflow includes build verification, standards compliance checks, and cross-component consistency validation.

If you have a code change in mind, please open a feature request or bug report describing the desired outcome. The maintainers will use coding agents to implement it in a way that aligns with the project's standards.
