## Decisions captured

| Decision | Outcome |
|---|---|
| Theme tokens — `--bs-*` (spec) vs `--theme-*` (project standard) | `--theme-*` with SCSS-variable fallbacks per `custom.scss` precedent. `--theme-metric-positive` / `--theme-metric-negative` added to light-mode `:root` (were dark-only). |
| Sparkline | Added (was not in the spec). MASTER_COMPONENT_LIST.md §10.2 explicitly called for one and the library should be a superset of any single app's needs. |
| Click semantics | `href` → `<a>`, `onClick`-only → `<button>`, neither → `<div>`. Native semantics; no `role="link"` on a div with manual key handlers. |
| Slug | New `metriccard` component, not an extension of the `.metric-card` Bootstrap utility class — lifecycle states + handle make it a real component, not a CSS utility. |
| Layout-stability test in jsdom | Asserts the structural invariant (`.metriccard-value` always present in DOM, height reserved via `min-height`) since jsdom doesn't compute layout. |

## Trend intent override

Decoupled from arrow direction so semantics are independent. Examples:
- Errors increased: `direction: "up", intent: "negative"` (red up-arrow).
- Costs avoided: `direction: "down", intent: "positive"` (green down-arrow).

## Open / parked

- Real-time pulse animation on the value when state transitions ready→ready with new data — not requested; could be added with a CSS class toggled in `setValue`.
- Multi-metric variant (two values stacked) — out of scope per spec.
